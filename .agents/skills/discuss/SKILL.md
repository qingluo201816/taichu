# Discuss 技能

## 核心目标

把模糊需求变成可执行决策。你的职责不是立刻实现，而是帮助用户看清选择、代价、风险和后续扩展影响。

## 触发边界

- 只有当前用户消息包含“讨论”两个字，或用户显式调用 `/discuss` 时，才进入本技能。
- 如果发现是误触发，且用户没有表达想进入讨论流程，立即停止使用本技能，按普通任务继续。
- 本技能适用于高影响决策，不适用于纯信息查询、明确 bug 修复、用户已经给出完整实施方案的任务。

## 工作原则

- 用中文输出，语言清楚直接，让用户能看懂每个取舍。
- 先收敛问题，再给方案；不要用一堆选项把用户推回迷雾里。
- 默认给 2-3 个互斥方案；每个方案必须说明收益、代价、风险和扩展影响。
- 必须给出一个推荐方案，并说明为什么它更适合当前上下文。
- 只在缺失信息会改变决策时提问；能合理推断时，先说明假设并继续。
- 不写代码、不改文件、不执行变更操作，直到用户明确选择方案或说“按你推荐来”。
- 如果用户确认方案，退出讨论模式，按普通执行流程实施。

## 上下文获取

在讨论前获取足够上下文，但保持克制：

- 涉及项目规范、架构或代码落位时，读取必要的本地规则、架构文档和相关代码。
- 涉及外部事实、市场现状、框架能力、最佳实践或“市面上类似方案”时，必须搜索并优先参考官方文档或一手资料。
- 不确定来源可靠性时，明确标注“这是推断”或“这是基于资料的建议”。

## 讨论流程

1. **复述目标**：用 1-2 句话说明你理解的目标和决策范围。
2. **列出约束**：列出已经确定的硬约束、项目原则、不可变决策和现实限制。
3. **补关键问题**：如果缺失信息会影响方向，只问 1-3 个短问题；不要开放式追问一长串。
4. **给出选项**：提供 2-3 个互斥方案，并按“适合场景 / 收益 / 代价 / 风险 / 扩展影响”说明。
5. **推荐方案**：明确推荐一个方案，说明它为什么最符合长期可维护性、职责边界和当前目标。
6. **等待确认**：请用户选择方案，或确认“按推荐方案执行”。
7. **沉淀结论**：用户确认后，用简短结论记录“决定做什么 / 不做什么 / 落到哪里 / 怎么验证”。

## 选项交互

如果当前环境提供可用的用户输入或选项弹窗工具，优先用弹窗向用户展示 1-3 个互斥选项：

- 推荐项必须排第一，并在标签里标注“推荐”。
- 每个选项用一句话说明影响或取舍。
- 只有当问题非阻塞、且存在安全默认项时，才允许设置自动继续；否则必须等待用户选择。

如果没有弹窗工具，就在正文中使用这个格式：

```markdown
我理解的目标：
...

关键约束：
...

可选方案：
1. 方案 A（推荐）：...
2. 方案 B：...
3. 方案 C：...

我的建议：
...

请你确认：选 A、B、C，还是补充一个限制条件？
```

## 高质量标准

- 好的讨论不是“平均陈列”，而是把真正的分岔点指出来。
- 好的方案不是“现在最快”，而是解释未来新增功能时会改哪里、不会污染哪里。
- 好的风险提示必须具体到行为后果，例如“以后每加一个 Agent 都要改注册中心”。
- 好的推荐必须能落地成文件位置、接口边界、验证方式或实施步骤。
- 不要把个人偏好伪装成事实；依据不足时说明依据不足。

## 禁止事项

- 禁止在用户确认前修改文件、写代码、安装依赖、提交 Git 或执行迁移。
- 禁止提供 4 个以上方案，除非用户明确要求穷举。
- 禁止只说“各有优劣”而不给推荐。
- 禁止为了显得谨慎而反复追问低价值问题。
- 禁止把讨论写成论文；默认短而准，必要时再展开。
