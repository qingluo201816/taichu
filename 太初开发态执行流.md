# 太初开发态执行流

> 更新日期：2026-06-27

## 角色分工

人负责目标、约束、优先级和最终判断。

GPT Pro 负责澄清需求、拆任务、定验收标准、做 review。

Codex 负责进代码库、改代码、跑测试、提交结果。

太初项目级 Skills 负责约束 Codex 的开发态行为，避免每次任务都重新约定执行方式。

## 标准执行流

1. 用户在 GPT Pro 网页版讨论需求。
2. GPT Pro 输出 PRD、技术约束和给 Codex 的任务包。
3. 用户把任务包粘贴给 Codex。
4. Codex 触发 `taichu-task-package-execution`，按任务包执行。
5. Codex 在改代码前确认执行方案；用户已明确授权时可直接执行。
6. Codex 按最小可验证增量修改代码，不做无关重构。
7. Codex 运行测试、构建、启动或网页验证。
8. Codex 输出任务包执行报告。
9. Codex 自动触发 `taichu-git-publish`，只提交本次任务相关文件并推送远程仓库。
10. 用户将执行报告、commit hash 和分支信息复制给 GPT Pro review。
11. GPT Pro 基于远程最新代码或提交 hash 验收本次任务，并继续拆下一个任务。

## 手动验收摘要流

当用户需要单独整理验收材料时，触发 `taichu-acceptance-brief`。

该 Skill 输出：

- 本次任务目标
- 已实现功能
- 关键变更文件
- 自动化验证结果
- 网页手动验收路径
- GPT Pro 验收重点
- 已知风险或未完成项

如果用户只需要聊天内容，Codex 只在回复中给出摘要；如果用户明确要求生成文件，再创建文档。

## Git 发布流

Git 发布是任务包执行完成后的默认动作，除非用户明确要求本次不提交、不推送。

发布前 Codex 必须：

- 检查 `git status --short`
- 区分本次任务变更和用户已有改动
- 确认测试或验证结果
- 排除 `.env`、密钥、缓存、构建产物、虚拟环境和临时文件
- 只 stage 本次任务相关文件

如果测试失败、关键验证未完成且没有合理原因、发现敏感文件、或无法区分本次改动和用户已有改动，Codex 必须停止发布并说明原因。

发布后 Codex 必须报告：

- commit hash
- 当前分支
- 推送结果
- 已提交文件
- 剩余未提交文件
- 给 GPT Pro 的验收提示

## 太初项目约束

- Python 使用 `uv`，禁止 pip、poetry、pipenv。
- 前端位于 `web/`，使用 Next.js、shadcn/ui、Tailwind CSS。
- 后端使用 FastAPI。
- 系统只服务单本玄幻小说，不设计多小说管理、多租户或 `project_id`。
- `domain` 层保持技术无关。
- Agent 即插件，新增 Agent 应通过新增目录和实现协议完成。
- 修改启动关键文件后必须验证 `start.bat`，或说明无法验证的原因。

## 推荐触发语

- “按这个任务包执行”
- “执行完给我报告”
- “生成给 GPT Pro 的验收摘要”
- “把这次任务提交并推送”
- “总结网页怎么验证”
- “这次只执行不推送”

## 不做的事

- Codex 不负责重新讨论需求。
- Codex 不主动创建额外长期文档。
- Codex 不在风险未解除时提交或推送。
- Codex 不提交无法区分归属的工作区改动。
- Codex 不把未验证结果描述成已验收。
