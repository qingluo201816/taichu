# MVP 功能改版

> 更新日期：2026-06-30

## 文档目的

本文档用于沉淀 MVP 功能改版的当前产品契约。它不是最终实现说明，也不是接口冻结稿；它的作用是把写作区、正文编辑器、左侧大纲、AI 多轮对话记录、手动记录、收件箱、知识库、结构化知识卡和 LangGraph 的职责边界拆清楚，避免后续改代码时把不同层级的概念混在一起。

当前方向是：写作区保持轻量、即时、入口明确；复杂任务进入独立的智能体工作台。用户在写作区选择的是明确的任务入口和参考范围，AI 不再自行决定结果类型。右侧各 AI 入口本质上是特化的多轮对话：入口明确路由、调用特定 skill 或提示词设定，并把每轮真实发送给模型的提示词快照保存下来。

英文短词只用于后端字段、接口或代码标识。第一次出现时必须配中文解释。例如 `prompt_snapshot` 表示“提示词快照”，即本轮真正发送给模型的完整提示内容；`skill` 表示“技能或提示词能力模块”；`route` 表示“本轮处理路径”。

## 讨论与确认规则

后续讨论问题必须按连续题号呈现，并包含：原始题目、可选答案、关键名词解释、推荐选择、推荐理由。

作者只需要回复需要修改的题号和选项；没有回复的题目默认全部确认。示例：“7 改 B，12 改 A，其余确认”。

## 已确认的总原则

1. 写作区不为了展示框架能力而使用 LangGraph。简单生成、润色、总结和写作区特化多轮对话不强行走 LangGraph。
2. LangGraph 由任务入口触发，不由提示词触发。跨章节一致性检查、全文级剧情起伏判断、结构化事实抽取、证据链追踪、多步骤修订计划等复杂任务，未来放到智能体工作台。
3. 写作区永远不直接触发 LangGraph，只触发轻量服务。
4. 用户输入永远是主指令。选区内容、本章正文、全文参考都是上下文，不应反过来覆盖作者的当前请求。
5. 用户选择的是想要的任务入口，不是让 AI 自己判断结果类型。
6. 纯对话不是 AI 多轮对话记录，不保存，只是写作区里临时调用 AI 对话的便利入口。
7. 灵感和事实是作者手动写入的高频入口，AI 不参与思考。
8. 正文 Markdown 是一手原文来源；结构化知识库是作者确认后的核心事实；向量知识库服务复杂、模糊、跨章节召回。
9. AI 不能直接把候选事实写入结构化知识库。候选事实必须先进入待确认区，作者确认后才成为知识库事实。
10. 面向用户的按钮、状态、提示、功能说明必须是中文。后端字段、接口和代码标识可以使用英文，但不得直接当用户可见文案。

## 命名体系

后端、接口、测试和代码标识使用清晰语义命名，例如纯对话、续写、润色、设定、建议、证据、章节摘要、灵感、事实。前端允许提供一键切换的整套名词系统，切换只影响用户可见文案，不影响后端枚举和业务语义。

AI 在前端品牌化命名中对应“器灵”。AI 多轮对话记录在前端品牌化命名中对应“灵笺”。后端仍应使用 `ai_conversation`、`ai_task`、`conversation_message` 等清晰语义，不把“器灵”“灵笺”写成核心领域枚举。

| 后端语义名 | 默认中文名 | 品牌化名 |
|------------|------------|----------|
| 纯对话 | 纯对话 | 问灵 |
| 续写 | 续写 | 衍文 |
| 润色 | 润色 | 润笔 |
| 设定 | 设定 | 构界 |
| 建议 | 建议 | 策议 |
| 证据 | 证据 | 溯源 |
| 章节摘要 | 章节摘要 | 章要 |
| 灵感 | 灵感 | 灵引 |
| 事实 | 事实 | 事实簿 |

前端需要一个命名体系切换按钮，默认显示清晰名。切换按钮放在写作页右侧窄栏顶部。切换不持久化，刷新后回到默认清晰名。品牌化模式下，“AI”在用户可见文案中统一显示为“器灵”。

## 写作区整体布局

写作页右端保留很窄的一列入口按钮，按钮顺序为：纯对话、续写、润色、设定、建议、证据、章节摘要、灵感、事实。品牌化模式显示为：问灵、衍文、润笔、构界、策议、溯源、章要、灵引、事实簿。

右侧窄栏按钮默认显示图标加短文字。作者点击某个按钮后，展开该入口独有的功能面板。展开面板会压缩左侧编辑区，而不是覆盖编辑器。展开面板宽度第一版可拖拽调整。

同一时间只允许展开一个入口。每个入口各自暂存未发送内容；切换入口时旧入口未发送内容保留，直到切换章节或刷新页面。

入口展开后的面板自上而下分为三块：上方是正文参考和参考范围，中间是结果记录，底部是输入框。发送按钮文案第一版统一为“发送”。

| 入口 | 性质 | 是否 AI 生成 | 是否保存为 AI 多轮对话记录 |
|------|------|--------------|------------------------|
| 纯对话 | 临时对话 | 是 | 否 |
| 续写 | AI 任务 | 是 | 是 |
| 润色 | AI 任务组 | 是 | 是 |
| 设定 | AI 任务 | 是 | 是 |
| 建议 | AI 任务 | 是 | 是 |
| 证据 | AI 任务 | 是 | 是 |
| 章节摘要 | AI 任务 | 是 | 否，单独显示在左侧章节摘要区域 |
| 灵感 | 手动记录 | 否 | 否，进入 Inbox 灵感 |
| 事实 | 手动记录 | 否 | 否，进入 Inbox 待确认事实 |

## 正文编辑器契约

章节正文就是正文。顶部工具栏不需要“正文 / 标题一 / 标题二 / 引用”四个按钮，第一版从用户可见界面移除。章节标题由左侧章节名称映射展示，不在正文编辑区内编辑。

编辑器工具栏第一版需要优先提供：字体大小、字体样式、编辑页面背景、撤销、恢复、查找、替换、一键段落排版、保存状态、手动保存。

一键段落排版用于把当前章节正文按小说正文阅读习惯整理为自然段。第一版重点处理段落分隔、首行缩进或等价视觉缩进、连续空行保留策略、行距和段距。它是作者主动触发的操作，不自动改写正文内容。

正文底层必须保留作者输入的多行空格和多行换行。作者往正文中加入空格、空行或多行间隔后，刷新页面、切换章节、保存再打开，都不能被压缩成最多一个换行。存储层和 Markdown/Tiptap 转换层不得随意 `trim` 或合并作者正文中的有效空白。

正文编辑区可以展示章节标题，但标题只映射左侧章节名，不允许在正文编辑区上部直接编辑。章节标题的重命名入口只在左侧大纲中。

项目中不允许出现英文按钮、英文功能介绍或内部状态文案，例如 `NON_FACT` 这类内部枚举不得出现在用户界面。所有用户可见文案必须中文。

“导出与重建”不作为普通用户可见功能。导出功能第一版搁置；如果该入口影响写作体验，可以先从用户界面移除。底层重建能力可以作为开发或内部能力保留，但不能在首页或写作主流程里打扰作者。

## 左侧大纲与分卷章节契约

左侧大纲第一版应采用“分卷 → 章节”的树状结构。卷默认都显示卷名；章节列表只展开当前选中卷下的章节，避免左侧大纲过长。

左侧大纲需要提供新建卷、新建章节、重命名卷、重命名章节按钮。重命名章节只在左侧大纲完成，正文编辑区上方只展示映射后的章节标题，不提供标题编辑。

章节显示名称需要去掉无意义前导零。例如底层文件或旧数据可能是“第0001章 大田金鳞元神出”，前端显示应为“第1章 大田金鳞元神出”。这只是展示规范，不要求第一版强制改底层文件名。

新建章节时应归属当前卷。新建卷后默认展开该卷并允许继续新建章节。章节排序和卷排序后续需要能稳定保存，避免刷新后顺序丢失。

## 写作区保存契约

保存按钮是主保存行为。作者点击保存，表示正式写入章节 Markdown 文件。`Ctrl+S` 与点击保存按钮完全等同。保存按钮鼠标悬停时提示快捷键 `Ctrl+S`。

保存状态只保留：未保存、保存中、已保存、保存失败。“已保存 14:32”显示最近一次成功保存时间，包含自动保存。

自动保存只做兜底，不制造存在感：间隔 5 分钟；只有内容有修改时才触发；成功不弹提示，只更新最近保存时间；失败时显示“自动保存失败，请手动保存”；失败后下一轮继续尝试。

切换章节、返回上一层时，如果有未保存内容，先保存。保存成功后再离开当前章节；保存失败则不切走，显示保存失败状态，并允许作者再次点击保存。浏览器关闭或刷新时，第一版只使用原生未保存提示。

## 输入与参考范围

写作区 AI 输入必须区分两类内容：作者待发送内容、正文参考内容。

作者待发送内容是作者这次真正想问、想改、想生成的请求。正文参考内容来自选区、本章或全文，AI 知道它是正文或资料，不把它当成作者指令。

参考范围应成为后端字段和路由策略，同时前端在每个 AI 面板里显示参考范围控件。第一版参考范围是单选。

| 参考范围 | 用户含义 | 初期路由 |
|----------|----------|----------|
| 选区 | AI 主要处理作者选中的文字，选区是强参考 | 选区 + 本章上下文，选区权重最高 |
| 本章 | AI 参考当前章节整体上下文 | 读取当前章节正文 |
| 全文 | AI 参考已沉淀的全书信息 | 写作区初期只查结构化知识库和检索层 |

没有选中文本时，选区类任务自动退化为本章范围。全文参考的多轮对话记录需要显示参考来源，至少显示知识卡或检索出处。纯对话默认完全不带小说上下文。

## 各入口参考范围审核表

| 入口 | 默认参考范围 | 可选参考范围 | 不开放范围 | 说明 |
|------|--------------|--------------|------------|------|
| 纯对话 | 无小说上下文 | 无 | 选区、本章、全文 | 纯对话不走小说上下文，作者可手动粘贴内容 |
| 续写 | 选区；无选区时本章 | 选区、本章 | 全文 | 续写有时需要参考本章；目标字数可选填 |
| 润色 | 选区；无选区时本章 | 选区、本章 | 全文 | 润色下含扩写、缩写、改写 |
| 设定 | 本章 | 选区、本章、全文 | 无 | 全文用于查结构化知识和检索层 |
| 建议 | 本章 | 选区、本章、全文 | 无 | 建议分小路由，默认看本章整体 |
| 证据 | 全文 | 本章、全文 | 选区 | 用于追问真相和依据，默认查知识库和检索层 |
| 章节摘要 | 本章 | 本章 | 选区、全文 | 第一阶段只总结当前章节 |
| 灵感 | 当前章节来源 | 不适用 | 不适用 | 手动记录，不是 AI 任务 |
| 事实 | 当前章节来源 | 不适用 | 不适用 | 手动记录，不是 AI 任务 |

## AI 任务入口

后续应使用一个更明确的任务类型枚举替代现有 `AIResultCardType + AIWorkflow` 双字段。

| 入口 | 当前理解 | 作者要求输入 | 默认参考范围 | 输出形态 |
|------|----------|--------------|--------------|----------|
| 续写 | 基于上下文生成后续正文 | 可为空，目标字数可选填 | 选区；无选区时本章 | 正文候选，只返回正文 |
| 润色 | 对已有文字做扩写、缩写、改写 | 可为空 | 选区；无选区时本章 | 正文候选，只返回正文 |
| 设定 | 补充世界观、人设、背景、规则等 | 必填 | 本章 | 固定结构设定结果 |
| 建议 | 给出修改建议、方向建议、问题判断 | 有 | 本章 | 结构化建议结果 |
| 证据 | 回答真实性、出处、前文依据 | 必填 | 全文 | 固定结构证据结果 |

章节摘要没有作者要求输入框。点击后按模板生成章节摘要。“选区提问”不再作为主要工作流概念。选区是参考范围，不是任务类型。

润色入口展开后包含三个子功能：扩写、缩写、改写。第一版不设置“普通润色”子项。

设定结果固定结构为“设定补充 / 使用建议 / 可能影响”。建议结果必须结构化为“问题 / 判断 / 建议”。证据结果固定结构为“结论 / 依据 / 推测 / 未确认点”。

## 右侧面板字段与文案

上方“选取输入和参考范围”区域标题确认用“正文参考”。

没有选区且默认本章时，顶部显示章节标题，例如“当前参考：本章 · 第三章”。这样作者能确认 AI 正在参考哪一章。

底部输入框占位文案按入口变化。占位文案是浅色提示文字，作者开始输入后自动消失。具体文案第一版由实现先定，后续可改，不再逐条让作者确认。初始建议如下：

| 入口 | 占位文案建议 |
|------|--------------|
| 纯对话 | 输入你想临时询问的内容 |
| 续写 | 输入续写要求，可为空 |
| 润色 | 输入扩写、缩写或改写要求，可为空 |
| 设定 | 输入你想补充的设定方向 |
| 建议 | 输入你想判断或改进的问题 |
| 证据 | 输入你想追问的依据或出处 |
| 灵感 | 记下一条灵感 |
| 事实 | 记下一条可能入库的事实 |

参考范围控件不记住作者临时选择。每次打开按入口默认值初始化，减少隐性状态，便于调试。

## 手动记录入口

灵感和事实是作者手动写入入口，AI 不参与思考。

灵感入口用于作者快速记录创意点子。字段第一版为：内容、优先级、关联章节、状态，不需要标题。灵感提交后进入 Inbox 灵感，不保存为 AI 多轮对话记录。

事实入口用于作者快速记录认为未来可能进入知识库的内容。UI 名称使用“事实”，不使用“待确认事实”作为入口名。字段第一版为：标题、内容、优先级、关联章节、出处、状态。事实提交后进入 Inbox 待确认事实，不保存为 AI 多轮对话记录。

证据结果不是事实入口。证据结果回答真相和依据；事实入口记录作者准备沉淀的内容。

## AI 多轮对话记录与消息机制

AI 多轮对话记录是写作区 AI 任务的存储单元。前端品牌化名词中可称为“灵笺”。纯对话、灵感、事实、章节摘要不进入普通 AI 多轮对话记录。

后端模型名确认使用 `AIWorkspaceConversation`，中文含义为“写作区 AI 对话”。这个名字明确它属于写作区右侧入口，不和未来智能体工作台、普通 Agent 对话或纯对话混淆。

不同功能按钮可以随时切换，因为每个按钮展开的是不同功能的对话区。但同一条 `AIWorkspaceConversation` 固定一个功能入口，也就是固定一个 `task_type`。一条续写对话不能在内部变成设定对话；如果作者要换功能入口，就切换到对应按钮下的对话区。

每个 AI 入口展开后，展示本入口在当前章节下的特化多轮对话记录。面板右上角第一版只需要提供两个对话级动作：“查找对话”和“新对话”。

AI 多轮对话记录第一版不在后端保存 `display_title`，即“展示标题”。前端可按任务类型、生成时间和内容临时生成展示标题。AI 多轮对话记录不保存品牌化名称，只保存后端语义类型。

所有写作区 AI 多轮对话记录必须有 `chapter_id`，即“章节标识”，归属当前章节。第一版不保存原始选区位置，只保存选区文本和正文参考内容。记录需要 `created_at` 和 `updated_at`，即“创建时间”和“更新时间”。第一版保存 `model_name`，即“模型名称”。

第一版不设计 AI 对话生命周期状态，不再保留 `discarded`。AI 对话只作为正常对话记录存在；作者不满意时可以新建对话，或继续在当前对话输入要求。

对话消息使用 `messages` 保存，每条消息至少包含 `role` 和 `content`。`role` 是“消息角色”，例如作者、助手、错误；`content` 是“消息内容”。

每次作者发送消息并触发模型调用时，都在触发模型调用的 `user` 消息上保存 `prompt_snapshot`。`prompt_snapshot` 表示“提示词快照”。第一版同时保存结构化对象和最终字符串。

`task_type` 表示“任务类型”；`subtask_type` 表示“子任务类型”；`reference_scope` 表示“参考范围”。对话根字段保存任务类型、子任务类型和参考范围，触发模型调用的 user 消息也保存本轮实际值。第一版一条对话内固定功能入口和参考范围；如果参考范围不合适，作者可以在输入框里补充说明，或新建对话。

`skill` 表示“技能或提示词能力模块”，`route` 表示“本轮处理路径”。第一版每轮都保存这两个字段。

生成失败写入 `messages`，使用 `role = "error"` 的错误消息记录。生成中允许切换到其他入口，后台继续生成，结果返回后仍写回触发时的入口和对话。同一个对话生成中不允许再次点击“发送”，发送按钮禁用。

来源引用使用双层保存：对话根字段汇总本对话用到过的全部来源，具体助手消息也保存本轮实际使用的来源。这样历史对话页面可以快速筛选“是否包含来源”，进入详情后也能看到每一轮回答具体依据了哪些章节、知识卡或检索结果。

续写结果只保存正文文本。润色结果只保存结果，不保存原文。润色的扩写、缩写、改写使用同一数据结构，通过子类型字段区分。

第一版不提供“复制”按钮。作者需要采用结果时，可以手动选择结果文本复制到正文或功能区。

## 查找对话与历史对话页面

写作区每个 AI 功能入口展开后，右上角提供“查找对话”和“新对话”。这里的“查找对话”只查当前章节、当前功能入口下的历史对话。例如作者正在第 3 章的“续写”入口内查找，只显示第 3 章的续写对话，不混入第 3 章的设定、建议、证据对话，也不混入其他章节。

写作区内查找的目标是快速回到当前写作现场的同类对话，不承担全局历史管理功能。查找结果建议用右上角按钮打开浮层列表，不挤占中间结果记录区。中间结果记录区默认只显示当前对话。

“新对话”按钮位于面板右上角。第一版点击后直接新建对话，不弹确认。新对话会清空当前未发送输入，不保留旧输入。

新对话不继承当前选区输入和参考范围。它按当前功能入口的默认参考范围重新初始化。例如续写入口重新按“有选区则选区、无选区则本章”的默认规则初始化；设定、建议、证据按各自入口默认范围初始化。

第一版新增单独“历史对话”页面，用于查看所有写作区 AI 多轮对话记录。该页面可以按章节、功能入口查看历史对话，前期重点服务于观察生成效果、复盘提示词快照和检查不同功能入口的输出质量。历史对话页面不是写作区右侧面板的替代品，也不承担编辑正文的职责。

历史对话页面第一版需要打开单条对话详情。列表点击后在同页右侧显示详情，不进入单独详情页。详情中默认隐藏提示词快照，点击“查看提示词快照”后展开。

历史对话页面第一版筛选条件为：章节、功能入口、是否有错误、是否有来源。第一版不允许删除对话，不允许重新运行某一轮对话。

## 创作收件箱

现有 Inbox 四类不再作为目标契约。后续方向是三类：灵感、待确认事实、待处理问题。

三类不强行共用同一套业务字段，但第一版都需要：状态、优先级、来源章节。状态统一为“待处理 / 已处理 / 已废弃”。已处理默认隐藏，只看待处理；已废弃不可恢复，允许查看详情，但不允许恢复和编辑。

灵感支持在 Inbox 手动新建，也支持从写作区灵感入口新建。待确认事实支持在 Inbox 手动新建，也支持从写作区事实入口新建。待处理问题第一版只在 Inbox 新建，不在写作区提供快捷入口。

三类 Inbox 项默认动作统一为：编辑、标记已处理、废弃。Inbox 第一版不做按章节筛选，不做站内搜索。现有 `saved_ai_cards` 这类“保存的 AI 卡片”收件箱通道应放弃。

## 章节摘要

章节摘要第一阶段只生成摘要正文，不生成关键事件、人物变化、伏笔候选、下一章钩子，也不生成待确认事实。

章节摘要模板为多段结构，但只包含摘要正文。第一版段落为：本章概述、人物变化、剧情推进。

同一章节多次总结时，覆盖旧摘要，不保留历史。覆盖前不需要确认，点击生成就覆盖。章节摘要不需要确认状态。

章节摘要展示在写作页左侧大纲树下方，独立于右侧 AI 多轮对话记录。

只要作者没点过生成摘要，不管当前章节是否为空，左侧章节摘要区域都显示“本章暂未生成摘要”。生成动作由右侧“章节摘要 / 章要”入口触发。

当前章节正文为空时，不触发摘要生成，提示“当前章节为空，无法生成摘要”。这种情况不算已生成过，左侧仍显示“本章暂未生成摘要”。

摘要生成中允许切换章节，生成结果归属触发时的章节，即使作者切走也继续写回原章节。

章节摘要生成中提示按当前命名体系切换：清晰名模式显示“正在生成章节摘要……”，品牌化模式显示“器灵正在整理章要……”。生成失败时保留旧摘要，并显示失败提示。

章节摘要第一阶段不做“转事实”。未来章节摘要后的事实抽取是第一个真正适合 LangGraph 的任务方向。

## 知识库与事实确认

知识库分三层：正文 Markdown、结构化知识库、向量知识库。

写作区的全文参考初期不是重型全书分析，而是让 AI 查结构化知识和检索层。

待确认事实确认进入知识库时，必须选择知识库分类，并按照分类规则校验后才能进入事实。确认知识时必须保留出处，没有出处不能确认。

知识库分类第一版保留现有分类：角色、境界、功法、地点、势力、物品、规则、事件、伏笔。

知识库冲突时阻止写入，并提醒作者处理。冲突提示需要显示冲突知识卡名称和摘要。分类校验失败时，在表单内显示中文错误提示，不使用弹窗。

别名字段不对所有知识卡统一开放。第一版角色、势力、地点、物品需要别名；规则、事件、伏笔、境界、功法不需要别名。

待确认事实确认入知识库前允许编辑内容。知识库确认直接复用 Inbox 详情面板，不进入单独确认页面。待确认事实确认入知识库前不强制要求手动填写标题。如果标题为空，可以根据内容自动生成。知识库确认成功后，原待确认事实状态变为已处理。

Inbox 待确认事实确认入库前需要预览最终知识卡。知识库确认入库成功后弹出选择：“查看知识卡 / 留在 Inbox”。

## 知识库结构化卡片与编辑页面

第一版必须针对结构化 JSON 设计角色卡、地点卡等知识库卡片。作者未来主要在网页中查看和编辑，不直接读 JSON 文件；JSON 是事实源存储格式，前端页面是主要操作入口。

知识库卡片采用“公共字段 + 类型专属字段 + 来源引用”的结构。公共字段保证列表、筛选、搜索和基础展示统一；类型专属字段保证角色、地点、势力、境界等信息不被压成一段杂乱文本。

公共字段为：`id`、`type`、`name`、`aliases`、`summary`、`body`、`tags`、`importance`、`status`、`source_refs`、`created_at`、`updated_at`。其中 `body` 保留为长正文补充区，`type` 是知识类型，`source_refs` 是来源引用，必须能追溯到章节、作者说明或其他确认来源。

第一版知识类型保留：角色、境界、功法、地点、势力、物品、规则、事件、伏笔。不同类型卡片允许拥有不同字段，不要求所有卡片字段一致。

知识库页面第一版需要全文搜索，搜索范围包括名称、别名、摘要、正文补充。这里的全文搜索只搜索知识卡字段，不搜索整本小说正文。

知识卡详情编辑第一版使用保存按钮，不使用自动保存。知识卡属于确认事实源，修改后会影响 AI 检索和设定一致性，必须由作者明确保存。

知识库页面允许直接新建知识卡，也允许从 Inbox 待确认事实确认入库。手动新建知识卡时来源可以为空，但确认入库或标记为有效知识前必须补来源。

知识卡第一版不物理删除，允许标记废弃。废弃卡保留记录但不作为有效知识使用，便于追溯。

知识库页面第一版布局为：左侧类型导航和筛选，中间知识卡列表，右侧详情编辑区。列表用于快速浏览和搜索，详情区用于结构化编辑，不能只展示原始 JSON。

知识库页面列表态展示：名称、类型、摘要、重要程度、来源数量、更新时间。详情态按类型展示字段分区，并保留来源引用区。编辑时必须使用中文字段名，不直接暴露 `source_refs`、`aliases` 等内部字段名；内部字段可以保留英文。

角色卡第一版支持：身份、重要程度、所属势力、当前境界、功法、物品、关系摘要、秘密摘要、出场章节、状态记录。角色通过 `importance` 区分核心角色、重要配角、普通角色。关系字段第一版先用关系摘要文本，不做结构化关系列表。

角色状态记录第一版需要结构化字段，按章节记录境界、位置、生死、阵营等字段。角色的境界、位置、生死、阵营、认知秘密等会随章节变化，是玄幻一致性检查的重要基础。

地点卡第一版支持：所属区域、上级地点、控制势力、地点规则、重要资源、危险程度、出现章节、关联事件。地点卡需要层级字段，例如上级地点、所属区域。

势力卡第一版支持：势力类型、首领、驻地、势力范围、敌友关系、核心规则、重要成员、关联地点。

境界卡第一版支持：境界体系、排序、能力边界、突破条件、典型表现、代价或限制。境界卡是玄幻一致性检查的重要基础。

功法卡第一版支持：功法类型、修炼条件、能力效果、限制代价、关联角色、关联境界、来源。

物品卡第一版支持：物品类型、持有者、能力、限制、来源、当前状态、关联事件。

规则卡第一版支持：规则内容、适用范围、限制条件、例外情况、关联设定。

事件卡第一版支持：发生章节、涉及角色、涉及地点、事件摘要、结果影响。

伏笔卡第一版支持：埋设章节、伏笔内容、关联角色或物品、当前状态、预计回收方向。

## 第一批实现接口边界

写作区 AI 任务接口拆成两个：创建对话接口、发送消息接口。

`AIWorkspaceConversation` 使用 `id` 作为唯一标识，不额外保存 `conversation_id`。接口路径中可以使用 `{conversation_id}` 作为路径参数名，但模型内部只保留 `id`。

`messages` 中每条消息需要独立 `message_id`，用于定位提示词快照、来源引用、错误消息和历史详情。

助手消息需要保存 `output_type`，例如正文候选、设定结果、建议结果、证据结果。写作区结果统一使用 `content` 字段，内部按 `output_type` 存不同结构。

创建对话接口路径使用 `POST /api/ai-workspace-conversations`。发送消息接口路径使用 `POST /api/ai-workspace-conversations/{id}/messages`。

创建对话接口用于“新对话”动作，负责建立一条新的 `AIWorkspaceConversation`。点击“新对话”后不立刻写入后端，只在前端创建临时状态；发送第一条消息时再写入，避免空对话污染后端数据。

创建对话接口不允许前端传入初始正文参考。正文参考在发送第一条消息时再传入，因为它只和触发生成的那一轮绑定。

发送消息接口用于在已有对话中追加作者消息、生成提示词快照、触发假输出或后续真实模型调用、写入助手消息。发送消息接口返回完整对话，前端不需要自行合并消息。

第一版假输出也必须生成完整提示词快照，完整走提示词构造链路。这样即使暂时不用真实模型，也能验证作者输入、正文参考、任务类型、参考范围和处理路径是否被正确组装。

第一版假输出按不同任务类型生成不同结构。续写、润色、设定、建议、证据分别返回不同结构。假输出前端显示“模拟输出”，避免误判为真实模型能力。

第一版提示词快照不限制长度，优先服务调试、复盘、重放和评测。后续如果出现存储压力或隐私问题，再讨论截断、冷存储或单独文件保存。

发送失败后保留作者输入框内容，方便重试或修改。发送成功后清空底部输入框。

README 暂不动，只更新本文档。当前细节仍在快速迭代，README 只在契约进一步稳定后同步总览。

## 任务结果结构

续写结果只保存一段正文文本。不要解释，不保存多版本候选，字数可由前端计算。

润色结果只保存润色后的正文，不保存原文，不保存修改说明。原文来自正文参考或提示词快照。

设定结果保存“设定补充 / 使用建议 / 可能影响”。它不是事实，不直接生成待确认事实。

建议结果保存“问题 / 判断 / 建议”。

证据结果保存“结论 / 依据 / 推测 / 未确认点”。其中“依据”可以为空，但必须明确“未找到明确证据”。

## 历史对话页面

历史对话页面路径使用 `/ai-history`。

历史对话列表第一版展示：时间、章节、功能入口、模型、是否错误、是否有来源、首条用户输入摘要。

## 纯对话边界

写作区纯对话需要保留页面内临时消息历史，只存在前端内存，刷新丢失，不写入后端。

纯对话不自动使用当前选区，也不自动带小说上下文。如果作者想问某段文字，需要手动粘贴到输入框。

历史对话页面不展示纯对话，因为纯对话不保存为写作区 AI 多轮对话记录。

## 功能入口打开状态

打开某个功能入口时，自动加载当前章节、当前入口最近一条对话。这样可以快速回到最近写作上下文。

如果当前章节当前入口没有历史对话，面板中间显示空状态和提示文案，不自动创建空对话。

新对话后，正文参考内容按当前入口默认规则重新生成，不完全清空，也不保留旧正文参考。作者可以手动编辑正文参考内容；手动编辑后的正文参考不回写正文，只影响本轮 AI。

事实入口提交后不允许直接确认入知识库。写作区事实入口只做快速记录，提交后进入 Inbox 待确认事实；确认入知识库在 Inbox 详情面板处理。

## LangGraph 使用边界

写作区中的轻量生成、特化多轮对话、润色、设定、建议、证据、纯对话、章节摘要第一阶段都不强行使用 LangGraph。

第一个真正 LangGraph 任务方向是“章节摘要后的事实抽取”。章节摘要先完成摘要，后续再由 LangGraph 做事实候选提取、结构化归类、证据关联和人工确认流转。

智能体工作台第一版不需要任务历史和可恢复状态，只运行完成后出结果。但未来一定要支持任务历史和可恢复状态。

## 与现有后端的主要冲突

1. `AIResultCardType` 和 `AIWorkflow` 职责重叠，应合并为一个面向用户任务意图的类型。
2. `SelectionMode.ASK` 容易把“选区”误当工作流，应改为任务类型 + 参考范围。
3. `AIResultCardStatus.INSERTED / SAVED_TO_INBOX / CONVERTED_TO_PENDING_FACT / DISCARDED / RETRIED / GENERATED` 不再作为 AI 对话生命周期状态。
4. `AICardService.mark_inserted` 后续应删除。
5. `AICardService.save_suggestion_as_idea` 的语义过窄，且建议结果不再保存为灵感。
6. `InboxSnapshot.saved_ai_cards` 应从目标契约中移除。
7. `POST /api/ai-cards/selection` 后续应改造成更通用的写作区 AI 任务入口。
8. `POST /api/agents/chat` 当前是普通服务调用并产出建议卡；后续写作区纯对话不应保存为 AI 多轮对话记录。
9. 现有章节摘要后端会生成结构化字段和候选事实，第一版目标应收敛为只生成摘要正文。
10. 旧的插入正文、保存到 Inbox、丢弃、重试状态相关测试可以直接删除重写，不做兼容。
11. 当前开发阶段允许破坏性清理旧 JSONL 数据，不做迁移。
12. 后端 API 允许破坏性改名，前后端同步改，不做旧接口兼容。
13. 第一批写作区 AI 对话后端模型使用 `AIWorkspaceConversation`，旧 AI 卡片模型不再作为目标契约。
14. 写作区内“查找对话”只查当前章节、当前功能入口，不做跨入口和跨章节搜索。
15. 第一版新增独立“历史对话”页面，用于按章节或功能入口查看所有写作区 AI 对话。
16. 写作区“新对话”直接新建，不确认、不继承当前选区输入和参考范围。
17. 第一批接口拆成创建对话接口和发送消息接口，不采用每个入口单独接口。
18. 第一版假输出也生成完整提示词快照，并按不同任务类型返回不同结构。
19. 不同功能按钮可以随时切换，但同一条写作区 AI 对话固定功能入口。
20. 一条写作区 AI 对话第一版固定参考范围；参考范围不合适时，作者通过输入补充说明或新建对话处理。
21. 来源引用同时保存在对话根字段汇总和具体助手消息本轮来源中。
22. 历史对话页面可查看单条详情，但第一版不允许删除、隐藏或重新运行对话。
23. 纯对话只保留前端临时消息历史，刷新丢失，不进入历史对话页面。
24. 知识库第一版必须面向结构化 JSON 卡片设计，并提供高效查看与编辑页面。
25. 正文编辑器必须保留多行空格和多行换行，移除不需要的标题/引用按钮，并提供一键段落排版。
