# MVP 迭代规则与 LLM 边界补充

> 更新日期：2026-06-30

## 文档目的

本文档用于补充《MVP 功能改版》中本轮新增的迭代规则和 LLM/RAG 边界。若本文与旧产品描述冲突，以本文和《MVP 功能改版》的较新结论为准。

## 迭代回复规则

每次产品契约迭代后，必须先同步本轮已确认和已修改的信息，再生成下一批新题目。

同步信息需要说明：本轮哪些题被修改、哪些题默认确认、文档是否已经更新、当前还有哪些关键边界没有定。

用户只需要回复需要修改的题号。没有回复的题目默认确认。后续不得重复追问已经默认确认的问题。

下一批问题必须一次多给一些，除非某个问题会明显卡住后续字段、页面、接口或状态流转设计。

每个问题必须交代前后文，不能只给题目和选项。固定结构为：前后文、原始题目、可选答案、关键名词解释、推荐选择、推荐理由。

## 第一版 LLM 边界

MVP 第一版不做真实 LLM 效果评测。

第一版目标是先把主功能搭起来，包括正文编辑器、分卷章节大纲、写作区右侧 AI 入口、AI 多轮对话记录、假输出链路、Inbox、知识库结构化卡片、知识库查看与编辑页面。

第一版可以保留 LLM 调用接口和提示词快照结构，但真实 LLM 接入、真实模型输出质量、提示词效果评测、模型对比、自动评分、生成质量评估都不作为本阶段目标。

假 LLM 的任务不是证明 AI 能力，而是打通前后端契约。假输出必须能验证：入口路由、作者输入、正文参考、参考范围、提示词快照、任务类型、输出结构、来源引用字段和历史记录展示是否正确。

第一版 AI 相关验收只验功能链路，不验真实回答质量。验收表述应使用“检查输出结构、链路和提示词快照”，不要写成“检查输出质量”“评测回答质量”或类似会被理解成真实 LLM 效果评测的描述。

历史对话页面、写作区 AI 面板和证据入口第一版用于检查输出结构、链路、来源字段和提示词快照，不用于判断模型回答好不好。

## 假 LLM 与真实 LLM 预留

第一版 AI 生成服务需要设计成可替换实现：同一套业务接口下，第一版挂假 LLM，后续替换真实 LLM。

假 LLM 只做固定模板输出，不模拟复杂检索、总结、归纳和真实回答能力。

假 LLM 输出时也必须生成提示词快照，用于验证未来真实模型会收到的作者输入、正文参考、参考范围、任务类型和处理路径。

假输出必须显示“模拟输出”标识。历史对话页面也需要显示该标识，避免以后回看时误以为这些记录代表真实模型效果。

第一版设置页不显示真实 LLM 配置入口。模型、API Key、服务地址等配置等真实 LLM 接入时再加。

## RAG 与检索增强边界

RAG 表示“检索增强生成”，即先从知识库或章节中检索相关材料，再把材料交给模型整理回答。

MVP 第一版不冻结 RAG 的最终方案，不要求实现真实的检索增强回答质量。

第一版需要先把 RAG 未来会用到的数据入口和接口边界留好：结构化知识库、来源引用、知识卡搜索、章节来源、AI 对话里的来源引用字段、证据入口的结果结构。

因为第一版使用假 LLM，不能真正验证“检索后再整理回答”的效果。因此本阶段只验证数据流和字段，不评估 RAG 回答是否聪明。

证据入口第一版保留，用假输出模拟“结论 / 依据 / 推测 / 未确认点”的结构，但不要求真正完成复杂检索、归纳和生成。

证据入口未来优先查结构化知识库，再查章节检索。第一版只保留这个路由意图和字段边界，不评估真实检索效果。

## 主导航与页面入口

MVP 第一版主导航包含：写作、知识库、Inbox、AI 历史、智能体工作台、设置。

主导航顺序为：写作、知识库、Inbox、AI 历史、智能体工作台、设置。

智能体工作台第一版先保留，并标记为实验功能。如果现有页面不影响写作区、知识库、Inbox、AI 历史和设置的主流程，就先做兼容保留；如果后续发现它干扰 MVP 主流程，再单独讨论收敛或隐藏。

写作是主工作台。知识库是结构化事实源。Inbox 处理灵感、待确认事实和待处理问题。AI 历史用于检查输出结构、链路、来源字段和提示词快照，不用于评测真实回答质量。设置第一版只放编辑器偏好和基础项，不显示真实 LLM 配置。

## 知识库设定入口

第一版必须有能添加知识库设定的入口。

入口至少包括两条路径：

1. 写作区“事实”入口：作者快速记录一条可能进入知识库的事实，提交后进入 Inbox 待确认事实。
2. 知识库页面直接新建：作者在知识库页面选择类型后，直接新建角色卡、地点卡、势力卡、境界卡、功法卡、物品卡、规则卡、事件卡或伏笔卡。

写作区事实入口强调快，不做复杂入库表单。提交后只进入 Inbox 待确认事实，不弹出知识卡类型选择。

写作区事实入口第一版字段为：标题、内容、来源章节、出处、优先级。状态由系统默认设为待处理，不作为作者必填项。

写作区灵感入口第一版字段为：内容、来源章节、优先级。灵感是创意记录，不等于事实，不进入知识库。

知识库页面直接新建强调结构化编辑。直接新建知识卡时必须先选择知识类型，新建后的默认状态为草稿。草稿不参与 AI 检索和证据回答。

草稿知识卡补齐来源后不会自动变为有效，必须由作者点击“标记为有效”。知识卡详情区第一版主按钮为：保存、标记为有效、标记废弃。

待确认事实确认入库时，需要预览最终知识卡；确认成功后，待确认事实保留并变为已处理，知识卡进入结构化知识库。

右侧“设定”入口生成的内容第一版不允许一键转成待确认事实。第一版不做真实 LLM，不能制造“AI 生成设定直接入库”的路径。作者需要手动通过“事实”入口或知识库页面新建来沉淀设定。

## 知识卡 JSON 与存储结构

知识卡 JSON 第一版采用“每张知识卡一个 JSON 文件”。

知识卡文件按类型分目录保存，例如角色卡放入角色目录，地点卡放入地点目录。目录命名可由实现按现有项目风格确定，但结构上必须与知识类型对应。

知识卡需要稳定标识字段，字段名统一使用 `id`，不使用 `card_id`。`id` 是知识卡内部稳定标识；名称只是展示字段，可以重命名。文档中可以用“知识卡 ID”描述概念，但 JSON 字段写作 `id`。

知识卡主名称字段统一使用 `name`。摘要字段统一使用 `summary`。知识卡标记为有效前，必须校验名称、摘要和来源引用。

知识卡字段中保留 `tags`，但第一版前端不展示标签，也不要求作者维护标签。

角色卡当前境界、角色卡所属势力、地点卡控制势力、功法卡关联境界第一版都先用自由文本，不强制引用境界卡或势力卡。后续一致性检查阶段再讨论结构化引用。

## 知识库搜索与来源引用

知识库页面作为一级页面出现在主导航里。

知识库搜索第一版只在当前选中的知识类型内搜索，不做默认全类型混排。跨类型搜索后续可以做，但必须清晰标注每条结果的类型。

知识库某类型为空时，显示“新建当前类型知识卡”的空状态。

草稿知识卡在当前类型列表中显示，并标记“草稿”。废弃知识卡默认隐藏，不进入常规“全部”列表；通过“只看废弃”单独查看。

知识库类型内状态筛选为：全部、草稿、有效。废弃默认隐藏，通过“只看废弃”单独查看。

来源引用结构第一版采用统一结构：来源类型、来源 ID、显示名称、原文摘录、备注。

来源引用类型支持章节、知识卡、作者说明、外部资料四种。实际使用可以先以章节、知识卡、作者说明为主，外部资料作为预留类型；外部资料来源第一版不在前端开放填写。

来源引用需要保存原文摘录，用于快速判断依据是否可信。原文摘录第一版限制在 300 字以内。

作者说明来源需要独立保存正文内容。作者说明本身就是来源文本，不能只写一句来源备注。

来源引用使用统一的“添加来源”弹窗或面板。章节来源从分卷章节树中选择。来源原文摘录支持从章节选区自动带入，也支持手动填写。

## 知识卡创建与页面交互

知识库页面的新建入口同时提供两种：左侧类型导航顶部有“新建知识卡”，点击后先选类型；当前类型列表顶部也有“新建当前类型”。

点击新建知识卡后不立刻写入后端，先在前端生成临时草稿，首次保存时再写入，避免产生空白草稿记录。

知识卡直接新建时默认状态为草稿。草稿不参与 AI 检索和证据回答。草稿补齐来源后不会自动变为有效，必须由作者点击“标记为有效”。

知识卡详情区第一版主按钮为：保存、标记为有效、标记废弃。第一版不物理删除知识卡。

## Inbox 与知识库入库

Inbox 第一版按三类分 Tab 展示：灵感、待确认事实、待处理问题。

待确认事实详情采用“左侧原事实，右侧知识卡预览”的布局。

待确认事实确认入库时，必须先选择知识卡类型，不使用 AI 自动猜测。

待确认事实确认入库后，原 Inbox 记录保留，状态变为已处理，不直接删除。

## 假输出结构与 AI 历史

证据入口第一版保留，用假输出模拟“结论 / 依据 / 推测 / 未确认点”的结构，并且必须包含模拟来源引用，用来测试前端来源展示链路。

续写假输出只返回正文候选文本，不返回解释说明。

设定假输出按“设定补充 / 使用建议 / 可能影响”三段展示。设定入口输出只是作者参考，不直接入库。

AI 历史详情中提示词快照默认折叠，需要调试时再展开。

AI 历史列表和详情都需要显示“模拟输出”标记，避免以后误判为真实模型结果。但“模拟输出”第一版只是列表和详情标记，不作为筛选条件。

AI 历史详情显示本轮任务类型、参考范围、输出类型、来源引用。

AI 历史第一版筛选包括：章节、功能入口、是否有来源、是否有错误。

AI 假输出测试重点验证字段和结构，不验证文本内容质量。重点字段包括：`task_type`、`reference_scope`、`prompt_snapshot`、`output_type`、`content`、`source_refs`。

任务包中需要提供每个 AI 入口的假输出示例，尤其是续写、设定、建议、证据等入口的固定模拟返回结构。

## 第一版验收边界

第一版 AI 相关验收只验功能链路，不验回答质量。

功能链路包括：入口可用、消息保存、提示词快照、假输出结构、来源字段、历史记录展示、模拟输出标识、知识库入口、待确认事实入库链路。

不得把第一版验收写成“真实 LLM 回答是否准确”“输出质量是否优秀”“RAG 效果是否好”。这些内容全部后置到真实 LLM 接入之后。

第一批实现优先做：编辑器、分卷章节、知识库、Inbox、假 AI 链路。真实 LLM 和 RAG 等骨架稳定后再接。

第一批后端测试重点覆盖：保存、空白保留、卷章大纲、知识卡创建/读取/更新/标记废弃、Inbox、假 AI 链路。不优先测试真实 LLM 输出。

下一步可以开始准备 Codex 执行任务包目录结构。任务包需要包含目标、阶段、停机点、验收和证据返回要求。

## 后续问题生成方向

下一批问题应优先覆盖：Codex 执行任务包目录结构、阶段划分、各阶段停机点、验收证据、知识卡 JSON 示例、假输出示例、第一批 API 请求响应、前端路由与导航状态。
